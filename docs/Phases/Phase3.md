# Phase 3 — Parallel Batch Renderer: Full Conversation to One Audio File

**Project:** RealTimeConApp
**Branch for this phase:** `phase3-batch-renderer` (off `main`)

## Goal

Take a real two-speaker script from two `.txt` files, parse it intelligently into ordered
turns, distribute those turns across 8 parallel CPU workers, generate each turn with the
correct cloned voice, and stitch everything back into one clean final `.wav` file — in the
right order, with natural pauses between speakers, sounding like a continuous conversation.
No GUI yet. No captions yet. Just: script in, finished audio out.

## What Phases 1 & 2 Already Confirmed (don't re-investigate)

- `torch.load` CPU patch works — model loads cleanly every time.
- Voice cloning via `audio_prompt_path` works for both speakers.
- Tuned settings per speaker documented in ARCHITECTURE.md — use them, don't re-tune.
- Real-time factor with cloning: **~8–10x RTF** for medium/long lines (the realistic range).
- Short lines (<5 words): outlier RTF of 14–23x — avoid generating them alone.
- Peak memory per worker: **~14MB Python-side** — 8 workers confirmed safe within 24GB RAM.
- Speaker 2 generates consistently faster than Speaker 1 at equivalent word counts.
- Model load time: ~8–13s — load once per worker at startup, reuse for all assigned turns.

## How the Three Modules Divide the Work

This phase fills in three files that have been stubs since Phase 0. Each has one job:

```
script_parser.py   — reads the two .txt files, builds ordered turn list, chunks smartly
orchestrator.py    — spins up worker pool, fans out chunks, collects results in order
audio_utils.py     — stitches ordered chunk audio files into one final .wav with pauses
```

These three must stay cleanly separated. `script_parser.py` knows nothing about workers
or audio. `orchestrator.py` knows nothing about file formats or stitching. `audio_utils.py`
knows nothing about speakers or scripts. Each can be tested and debugged in isolation.

## Script Format

Two plain `.txt` files in `scripts/` — one line per turn, no tags needed from the user:

```
# scripts/speaker1.txt
Did you catch the news today?
That bad, huh?
I thought we said we'd be good this week.

# scripts/speaker2.txt
Don't even get me started.
Honestly, yes. The whole thing was a mess from start to finish.
I know, I know. But here we are.
```

`script_parser.py` zips them together into an ordered turn list:
```python
[
  ("S1", "Did you catch the news today?"),
  ("S2", "Don't even get me started."),
  ("S1", "That bad, huh?"),
  ("S2", "Honestly, yes. The whole thing was a mess from start to finish."),
  ...
]
```

**Important:** turns are interleaved strictly in order — S1 line 1, S2 line 1, S1 line 2,
S2 line 2, and so on. If the two files have unequal line counts, the parser stops at the
shorter one and logs a warning — never silently drops turns.

## Smart Chunking Rules (baked in from Phase 2 findings)

`script_parser.py` chunks the turn list for parallel workers using these firm rules:

**Rule 1 — Turn is atomic.**
A turn is never split across chunks. Ever. The full text of one speaker's line stays
together in one chunk and is generated by one worker in one call.

**Rule 2 — Chunk by estimated generation time, not word count.**
Use Phase 2's confirmed RTF numbers to estimate each turn's generation cost:
- S1 medium (10–20 words): ~10.36x RTF baseline
- S2 medium (10–20 words): ~9.05x RTF baseline
- Long turns (30+ words): ~8x RTF for both
Target chunk duration: aim for roughly equal estimated generation time across all 8
chunks, not equal word counts — Speaker 2 is consistently faster, so naive word-count
balancing leaves S2 workers idle while S1 workers are still running.

**Rule 3 — Merge short turns.**
Any turn under 8 words gets merged with the next same-speaker turn before chunking,
producing one longer combined line rather than two short ones. This avoids the 14–23x
RTF penalty seen in Phase 2 for sub-5-word lines. Merged turns are joined with a
comma and a natural connective pause marker so the generated speech flows, not jarred.

**Rule 4 — Cut only at turn boundaries.**
When a chunk crosses its target duration threshold, the parser waits for the current
turn to finish completely, then cuts. Never mid-turn.

**Rule 5 — Respect speaker alternation across seams.**
Track which speaker ends each chunk. The next chunk must start on the other speaker.
If the natural cut point would produce same-speaker adjacency across a seam, the
parser extends the current chunk by one more turn to clear it.

**Rule 6 — Each chunk carries its own metadata.**
Every chunk handed to a worker includes: chunk index (for reassembly order), ordered
list of (speaker, text) turns, the reference clip paths for both voices, and the tuned
generation settings per speaker from ARCHITECTURE.md. Workers are self-contained —
they need nothing from each other.

## Parallel Worker Design

`orchestrator.py` uses Python's `multiprocessing.Pool` (not `threading`) — this matters
because torch's GIL behavior under threading is unreliable; `multiprocessing` gives each
worker a genuinely separate Python process with its own memory space.

**Worker startup:** each worker process loads the Chatterbox model once at init via
`initializer` / `initargs` in `Pool` — not once per turn. Model load time is ~8–13s,
paid once per worker at pool startup, then amortized across all turns in that chunk.

**Worker execution per turn:**
1. Call `tts_engine.generate(text, audio_prompt_path=ref_clip, **speaker_settings)`
2. Save output tensor as a numbered temp file: `output/chunk_{chunk_idx}_turn_{turn_idx}.wav`
3. Return the file path + chunk/turn index to the orchestrator

**Reassembly:** the orchestrator collects all returned paths, sorts them by
`(chunk_idx, turn_idx)` — not by arrival order, which is nondeterministic — then passes
the sorted list to `audio_utils.py` for stitching.

**Worker count:** default 8 (matches Ryzen 7 5700G core count). Exposed as a constant
in `config.py` so it can be tuned down if memory pressure is observed on first real run.

## Audio Stitching

`audio_utils.py` receives the sorted list of per-turn `.wav` paths and produces one
final output file. Stitching rules:

- Between turns of **different speakers**: insert a pause of `config.INTER_SPEAKER_PAUSE`
  (default: 0.4s — natural conversation gap, adjustable)
- Between turns of **same speaker** (if any consecutive same-speaker turns exist in the
  final ordering): insert `config.SAME_SPEAKER_PAUSE` (default: 0.15s — briefer, like
  someone continuing a thought)
- Final output: `output/conversation_final.wav` — 44100Hz, mono, 16-bit PCM, matching
  the format of the reference clips
